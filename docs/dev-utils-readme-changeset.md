# dev-utils

A collection of standalone developer tools built to work together or
independently. Part of **Project Crew** — a family of tools that follow
shared design principles and can be composed across projects.

Currently includes Python packages, bash utilities, and container
configurations.

---

## Project Crew

All tools in this collection follow the same conventions:

- src layout for Python packages
- Consistent return signatures (`{'success': bool, 'message': str, ...}`)
- Explicit exception hierarchies
- `encoding='utf-8'` on all file I/O
- Filenames use underscores, not hyphens
- No hard-coded values — config files or environment variables
- Each tool works standalone and can be imported by other tools

---

## Python Tools

**contactkit** — Multi-format contact import and CLI interface.
Parses CSV/VCF files from Gmail, Proton, Outlook, and Apple; inserts into a
contacts schema via dbkit; logs imports to a `contact_imports` audit table.
Gmail plugin complete; others in progress.

**dbkit** — Database abstraction layer.
Consistent connection and query interface for PostgreSQL (via psycopg3) and
SQLite. Credentials via `~/.pgpass` for PostgreSQL; rows returned as dicts.

**fletcher** — Raw GitHub URL manifest generator.
Reads `manifest.yml` files and outputs `manifest.fletch` files containing
raw GitHub URLs for every file in the manifest. Used by setupkit and
dr-filewalker.

**mcpkit** — MCP tool executor. *(Not yet ready for use.)*
Registers and dispatches tool calls via the Model Context Protocol. Plugin
architecture for adding new tool handlers.

**menukit** — YAML-driven CLI menu library.
Provides `MenuBuilder` and `InputPlugin` base classes for building consistent
interactive menus across CLI tools. Used by dr-filewalker and others.

**setupkit** — Plugin lifecycle manager.
Installs, updates, and checks the status of Python packages without manual
pip paths or version tracking. Reads package metadata from `manifest.fletch`
files generated by fletcher.

**sniffkit** — File type detection library.
Detects file types from content and extension. Classifier, detector plugins
(SQL, JSON, Markdown, HTML), and a looper for batch processing. Used by
dr-filewalker and the designing-gemma pipeline.

**todo** — JSON-backed task manager.
Standalone CLI tool; stores tasks in `~/.local/share/todo/todos.json` and
exports a `TODO.md` to the current working directory on exit.

**treekit** — Directory tree scaffolding from markdown.
Reads a markdown file containing a directory tree structure and creates the
corresponding empty files and directories on disk. Supports fenced code
blocks and bare tree text. Works as a standalone CLI or as a library.

**viewkit** — YAML-driven SQL view and query builder.
Defines and executes database views and queries from YAML configuration.
Includes an on-the-fly query runner for ad hoc use.

- `templates/` — starter templates for new scripts

## Containers

Docker Compose configurations in `containers/`. All containers follow
the same conventions: `.env` for local config, `127.0.0.1` binding
by default, `unless-stopped` restart policy.

### Database

- `adminer/` — lightweight database browser
- `pgadmin/` — PostgreSQL administration UI

### pygotty family

Browser-accessible terminals via GoTTY. Each container serves a
different context in a browser tab — no client installation required.

- `pygotty/` — Python 3.12 REPL proof-of-concept (port 8090)
- `pygotty/bash/` — persistent bash shell (port 8091)
- `pygotty/pygotty_files/` — bash + read-only host filesystem mount (port 8092)
- `pygotty/ollama_gotty/` — bash + Ollama local LLM backend (port 8093)

All pygotty containers can run simultaneously. See
`containers/pygotty/roadmap.md` for security hardening sequence.

### Development and testing

- `dev-utils-test/` — test environment for dev-utils toolchain
- `python-test/` — Python environment test container
- `setup-tests/` — setup script validation

## Installation

### Install a single package

```bash
cd python/<package>
pip install -e .
```

### Install via setupkit

Once setupkit is installed, use it to manage other packages:

```bash
setupkit install <package>
setupkit update <package>
setupkit status
```

### Bootstrap from scratch

```bash
bash setup.sh
```

## Using Individual Tools Without Cloning Everything

You don't need to clone the full repo. Git sparse checkout lets you
pull only the folder you need:

```bash
git clone --no-checkout https://github.com/carolynboyle/dev-utils.git
cd dev-utils
git sparse-checkout init --cone
git sparse-checkout set containers/pygotty
git checkout main
```

Replace `containers/pygotty` with any folder — `python/treekit`,
`python/fletcher`, `containers/adminer`, etc.

To get files without any git history at all:

```bash
svn export https://github.com/carolynboyle/dev-utils/trunk/containers/pygotty
```

This uses GitHub's SVN bridge — no SVN installation required beyond
the `svn` command itself.
